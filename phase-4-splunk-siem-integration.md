# Phase 4 – Splunk SIEM Integration

## Overview

The objective of Phase 4 was to deploy Splunk Enterprise as the Security Information and Event Management (SIEM) platform for the homelab, integrate endpoint telemetry generated by Sysmon, and validate that security events could be collected, searched, and analyzed through SPL (Search Processing Language).

This phase established the central logging platform that will be used throughout the remainder of the project for threat hunting, detection engineering, alerting, and incident investigation.

# Splunk Deployment

Splunk Enterprise was installed on the Windows Server 2022 virtual machine and configured as the central SIEM for the homelab.

Following installation:

- Splunk Web was successfully configured and made accessible through the local web interface.
- The Sysmon Operational Event Log was configured as a local Windows Event Log input.
- Endpoint telemetry was successfully ingested into the **main** index.
- Splunk services were validated after installation and configuration changes.

---

# Sysmon Telemetry Integration

Sysmon telemetry was collected from the following Windows Event Log channel:

```
Microsoft-Windows-Sysmon/Operational
```

Telemetry was successfully ingested into Splunk and verified through SPL searches.

Successful Event IDs included:

- Event ID 1 — Process Creation
- Event ID 4 — Sysmon Service State Change
- Event ID 8 — CreateRemoteThread
- Event ID 11 — File Create
- Event ID 13 — Registry Value Set
- Event ID 22 — DNS Query
- Event ID 255 — Sysmon Errors / Status

These events provide visibility into process execution, file creation, registry modifications, DNS activity, and additional endpoint behaviors commonly leveraged during security investigations.

# SPL Validation

The following SPL searches were used throughout deployment to validate telemetry collection and indexing.

Example searches included:

```spl
index=main
```

```spl
index=main | stats count by sourcetype
```

```spl
index=main sourcetype=xmlwineventlog
```

```spl
index=main sourcetype=xmlwineventlog
| xmlkv
| stats count by EventID
```

These searches confirmed successful ingestion of Sysmon XML telemetry and verified that endpoint events were searchable within Splunk.

# Event ID 3 Investigation

A significant portion of this phase involved investigating the absence of **Sysmon Event ID 3 (NetworkConnect)** telemetry.

Multiple validation steps were performed, including:

- Verification of Sysmon configuration
- Confirmation that the NetworkConnect rule existed within the Sysmon configuration
- Validation that Sysmon services were operating correctly
- Review of Sysmon operational status
- Verification of Windows Event Log channel availability
- Inspection of Sysmon XML telemetry
- Validation of Splunk ingestion
- Verification of XML parsing using `xmlkv`
- Review of Sysmon Event IDs successfully collected

Although multiple Sysmon Event IDs were successfully generated and ingested, NetworkConnect (Event ID 3) was not produced within this lab environment.

Based on the successful collection of all remaining endpoint telemetry, the issue was determined to be isolated to NetworkConnect generation rather than a failure of Splunk ingestion or Sysmon deployment.

Because the remaining telemetry pipeline was fully operational, the investigation was documented as a known lab limitation and the project proceeded to the next phase.
